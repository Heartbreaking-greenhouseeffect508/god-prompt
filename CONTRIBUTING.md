# Contributing to GodPrompt

GodPrompt is a production-grade engineering discipline framework. Contributions should reflect that standard.

## Before You Start

1. Read `core/00-THE-SKILL.md` — understand the protocol you're contributing to.
2. Check [existing issues](https://github.com/AKzar1el/god-prompt/issues) to avoid duplicate work.
3. For significant changes, open an issue first to discuss the approach.

## Submitting Issues

- Be specific. Include what you expected, what happened, and steps to reproduce.
- For protocol suggestions, explain the gap in the current system and provide a concrete example.
- One issue per problem. Don't bundle unrelated items.

## Pull Request Process

1. Fork the repo and create a feature branch from `main`.
2. Make your changes. Keep commits focused — one logical change per commit.
3. Follow the project's own standards:
   - **Understand before acting** — read the files you're modifying.
   - **Scope is sacred** — change only what's necessary.
   - **Evidence before claims** — test your changes and describe what you verified.
4. Update `CHANGELOG.md` if your change affects user-facing behavior.
5. Open a PR with a clear description of what changed and why.

## Code Style

- Match the existing voice: direct, precise, no filler.
- No AI slop words ("delve", "tapestry", "holistic", "synergy", "leverage").
- Markdown formatting should be consistent with existing files.
- Keep line lengths reasonable. Use blank lines to separate sections.

## What We're Looking For

- Protocol improvements backed by real-world usage.
- Bug fixes in path references, formatting, or broken examples.
- New anti-patterns discovered through practice.
- Landing page and documentation improvements.

## What We'll Reject

- Changes that add complexity without clear value.
- "While I'm here" scope creep in PRs.
- Generic content that doesn't match the project's voice.
- Unverified claims about what "should work."
